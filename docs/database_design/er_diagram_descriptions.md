# 数据库 E-R 图文字说明

在完成系统功能流程和数据库表结构设计后，需要进一步明确各业务实体之间的对应关系。E-R 图能够直观展示数据库实体、实体属性以及实体之间的一对一、一对多关系，为后续数据表设计、外键约束设计和业务数据流转提供依据。本系统围绕资产发现、资产分析、风险评估、机器学习漏洞情报分析和用户管理等功能建立数据模型，各模块之间以资产实体为核心进行关联。

## 数据库整体 E-R 图

数据库整体 E-R 图用于描述系统核心业务数据之间的总体关系。系统以资产发现任务为起点，发现任务在执行过程中会产生多条发现记录和验证记录，验证成功的目标会进一步写入资产基础信息表。资产实体保存域名、IP 地址、端口、页面标题、服务信息、首次发现时间和最近发现时间等基础属性，是后续资产分析、风险评估和漏洞情报分析的基础数据来源。通过整体 E-R 图可以看出，发现任务、发现记录、验证记录和资产之间形成了由扫描任务到资产入库的完整数据链路。

## 用户管理模块 E-R 图

用户管理模块 E-R 图主要描述管理员与用户账号之间的管理关系。系统中的管理员可以对多个用户账号进行新增、修改、启用、禁用和删除等维护操作，因此管理员与用户之间是一对多的管理关系。用户账号实体包含用户 ID、用户名、密码哈希、显示名称、角色、启用状态、创建时间、更新时间和最后登录时间等属性，用于支持系统登录认证、权限区分和账号状态管理。该 E-R 图体现了用户权限管理模块的数据组织方式，也为用户账号表的唯一约束和状态控制提供了设计依据。

## 资产发现模块 E-R 图

资产发现模块 E-R 图用于描述一次资产发现任务在数据库中的记录方式。每一次扫描会生成一条发现任务记录，任务记录中保存扫描域名、扫描模式、任务状态、扫描参数、结果摘要、开始时间和完成时间等信息。一个发现任务可以产生多条发现记录，用于保存不同阶段发现到的子域名、IP 地址、来源、证据文本、URL 列表、JS 端点列表和置信度等信息；同一任务也可以产生多条验证记录，用于保存探测后的端口、协议、状态码、标题、服务信息、成功标识和错误信息。通过该 E-R 图可以看出，资产发现模块采用任务、发现记录和验证记录分层保存的方式，既能保留扫描过程，也能支持后续结果追溯。

## 资产分析模块 E-R 图

资产分析模块 E-R 图以资产实体为中心，描述资产类型分析和资产标签之间的关系。系统对每个资产进行规则或模型分析后，会生成对应的资产分析结果，用于记录资产类型、模型置信度和分析时间；同时，系统还会根据规则命中情况为资产生成多个标签，标签信息包括标签名称、标签来源、置信度、命中规则和创建时间等内容。因此，资产与资产分析结果之间通常是一对一关系，资产与资产标签之间是一对多关系。该图说明了系统如何在资产基础信息之上扩展语义化分析结果，为后续风险评估提供输入。

## 风险评估模块 E-R 图

风险评估模块 E-R 图描述资产、资产分析、资产标签和风险评估结果之间的关系。系统在进行风险评估时，会综合资产基础信息、资产类型分析结果以及资产标签信息，按照规则评分方法生成风险分值、风险等级、评分详情和修复建议。一个资产对应一条最新的风险评估结果，而多个标签可以共同参与同一资产的风险评分过程。该 E-R 图体现了风险评估模块的数据依赖关系，即风险结论并不是单独生成的，而是由资产属性、分类结果和标签证据共同支撑。

## 机器学习漏洞情报分析模块 E-R 图

机器学习漏洞情报分析模块 E-R 图用于描述资产、CVE 知识库、严重等级模型和机器学习分析结果之间的关系。系统首先根据资产的域名、服务信息、标题和端口等特征识别可能关联的组件，再从 CVE 知识库中匹配相关漏洞信息，并结合严重等级模型对漏洞风险进行辅助分析。机器学习分析结果保存组件名称、组件置信度、匹配证据、匹配 CVE、严重性统计、风险分值、风险等级、薄弱点、分析说明、模型名称和更新时间等内容。该 E-R 图说明了机器学习模块并不直接替代规则评估，而是通过漏洞情报匹配和模型预测为资产风险判断提供补充依据。
